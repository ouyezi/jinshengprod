## 1. 概述

优化评委端评审结果的提交与查看策略，并增强管理端汇总页的可读性。

### 1.1 背景

当前实现中，评委提交后仍可通过 `load` 接口以只读方式查看已提交内容；且数据库 `(employee_id, reviewer_name)` 唯一约束阻止同一评委对同一员工重复提交。实际业务要求：

1. **提交前**：暂存 + 刷新可恢复草稿（保持不变）
2. **提交后**：评委端不可再查看已提交内容
3. **修改方式**：重新填写并提交，线下告知管理员保留哪一条
4. **管理端**：汇总页对重复提交、通过/未通过做颜色区分
5. **评委端**：当前状态区域展示生成结果规则说明

### 1.2 方案选择

采用 **方案 A：移除唯一约束 + 允许多条记录**。

| 方案 | 说明 | 结论 |
|------|------|------|
| A | 移除唯一约束，允许多条已提交记录并存 | ✅ 采用 |
| B | 保留唯一约束，覆盖式重提 | ❌ 丢失历史，不符合线下选留流程 |
| C | 版本号 + 历史表 | ❌ 改动过大，超出当前需求 |

## 2. 评委端行为

### 2.1 状态机（不变）

```
待提交 ←暂存─ 初始
待提交 ─生成结果→ 待确认 ─提交→ 已提交
```

### 2.2 各场景行为

| 场景 | 行为 |
|------|------|
| 提交前暂存 + 刷新 | 恢复草稿（不变） |
| 提交成功（同一会话，未刷新） | 页面保留刚填内容，表单只读，状态显示「已提交」；不触发 `load` |
| 刷新 / 重新选择同一员工 | 不加载已提交内容；显示空白表单 |
| 刷新 / 重选且存在已提交记录 | 显示 Alert 提示（见 §2.3） |
| 再次填写并提交 | 新建一条记录，不覆盖旧记录 |

### 2.3 已提交提示（Alert）

当 `load` 返回 `record=null` 且 `has_submitted=true` 时，在表单区域上方展示：

> 您已提交过该员工的评审，如需修改请重新填写并提交，线下告知管理员保留哪一条。

使用 Ant Design `Alert`，`type="info"`，`showIcon`。

### 2.4 当前状态 + 规则说明

在「当前状态：{status}」下方，**始终展示**生成结果规则说明（与当前分数/状态无关，作为操作指引）：

| 条件 | 行为 |
|------|------|
| 总分 ≤ 2 | 直接「不通过晋升」 |
| 总分 ≥ 4 | 直接「通过晋升」 |
| 2 < 总分 < 4 | 评委需要选择是否同意晋升 |

**UI 规格：**

- 位置：现有 `{status && <Text>当前状态：…</Text>}` 区块内，状态文字下方
- 样式：`Text type="secondary"`，字号与状态文字一致或略小
- 文案示例：

```
当前状态：待提交
生成结果规则：总分 ≤ 2 直接「不通过晋升」；总分 ≥ 4 直接「通过晋升」；2 < 总分 < 4 时评委需自行选择是否同意晋升。
```

- 已提交同会话只读时：规则说明仍展示（帮助评委理解当初流程）

### 2.5 按钮与只读

| 状态 | 暂存 | 生成结果 | 提交 | 清空重写 |
|------|------|----------|------|----------|
| 待提交 / 待确认 | ✅ | ✅ | 待确认时可提交 | ✅ |
| 已提交（同会话） | ❌ | ❌ | ❌ | ❌ |
| 已提交后再进入（空白表单） | ✅ | ✅ | 待确认后可提交 | ✅ |

## 3. 后端变更

### 3.1 数据库迁移

删除 `evaluation_record` 表上的唯一约束 `uq_employee_reviewer`。

```sql
-- SQLite 需重建表或 ALTER 方式移除约束（按项目现有 migration 惯例执行）
```

迁移后，同一 `(employee_id, reviewer_name)` 可存在多条记录。

### 3.2 `load` API

**请求：** `GET /api/evaluations/load?employee_id=&reviewer_name=`

**响应扩展：**

```json
{
  "employee": { ... },
  "record": { ... } | null,
  "has_submitted": true | false
}
```

| 字段 | 规则 |
|------|------|
| `record` | 仅当存在 `status ∈ {待提交, 待确认}` 时返回；取 `update_time` 最新的一条 |
| `has_submitted` | 是否存在任意 `status = 已提交` 的记录（同一 employee_id + reviewer_name） |

**不再**在 `load` 中返回已提交记录。

### 3.3 `upsert_draft` / `generate_result`

查找逻辑改为只匹配 **活跃记录**（`status ∈ {待提交, 待确认}`）：

- 存在活跃记录 → 更新该记录
- 不存在活跃记录（含仅有已提交记录的情况）→ **INSERT 新行**

移除 `ValueError("该评审已提交，不可修改")` 拦截。

若异常情况下存在多条活跃记录，取 `update_time` 最新的一条操作。

### 3.4 `submit` API

逻辑不变：仅 `status = 待确认` 可提交 → `已提交`。

### 3.5 `query_summary` API

响应每行增加 `id` 字段（记录主键），供前端 `rowKey` 使用。

## 4. 管理端汇总页

### 4.1 展示策略

全部记录展示，不做默认过滤。同一评委对同一员工的多条记录均可见。

### 4.2 行样式规则

按 `(评审对象姓名, 评委姓名)` 分组统计条数：

| 优先级 | 条件 | 行背景 | 附加标记 |
|--------|------|--------|----------|
| 1（最高） | 同组 ≥ 2 条 | `#fff7e6`（浅橙） | `Tag color="orange"` 文案「重复提交」 |
| 2 | 同组仅 1 条且 `评委确认结果 = 通过晋升` | `#f6ffed`（浅绿） | 无 |
| 3 | 同组仅 1 条且 `评委确认结果 = 不通过晋升` | `#fff1f0`（浅红） | 无 |
| 4 | 其他（草稿等） | 默认白色 | 无 |

**重复提交优先：** 当同组有多条时，忽略通过/未通过配色，统一使用重复提交高亮。

实现方式：Table `onRow` 返回 `style: { background }`；「重复提交」Tag 加在「状态」或「评审对象姓名」列。

### 4.3 Excel 导出

保持纯数据导出，不加行颜色或 Tag（与现有 `build_summary_export` 行为一致）。

### 4.4 rowKey

改为 `String(record.id)`，避免同一人多条记录 key 冲突。

## 5. 错误处理

| 场景 | 处理 |
|------|------|
| 同时存在活跃草稿 + 已提交记录 | `load` 返回活跃草稿；`has_submitted=true` 但不显示 Alert（因有草稿可继续编辑） |
| 多条活跃记录（异常） | 取 `update_time` 最新 |
| 提交后同会话误操作 | 按钮 disabled，不变 |
| 重提后管理端多条记录 | 管理员线下决定保留哪条，系统不做自动合并/删除 |

**Alert 展示条件（精确）：** `has_submitted === true && record === null`

## 6. 测试要点

### 6.1 评委端

1. 暂存 → 刷新 → 草稿恢复
2. 生成结果 → 提交 → 同会话内容可见、只读
3. 提交后刷新 → 空白表单 + Alert
4. 重填 → 暂存 → 生成 → 提交 → 成功（新记录）
5. 当前状态区域始终显示规则说明

### 6.2 后端

1. `load` 不返回已提交记录
2. `has_submitted` 在仅有已提交记录时为 `true`
3. 已提交后 `upsert_draft` 创建新行而非报错
4. 唯一约束已移除，同 pair 可有多条已提交记录

### 6.3 管理端

1. 重复提交行橙色高亮 + Tag
2. 单条通过/未通过绿/红背景
3. 重复组内不使用绿/红背景
4. `rowKey` 不冲突
